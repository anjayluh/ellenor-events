# EECS System Design

## Tenancy Model

Every domain object belongs to a `project_id`. Backend services must never rely on frontend filtering for tenant isolation.

## Identity Model

`users` stores global identity only. Roles belong to `project_members`, allowing one user to be an owner in one event and a viewer in another.

## Participants

Participants are not users. Family members, guests, and vendors can remain in `participants` without authentication.

## Budget Security

Budget responses are shaped by `budget_visibility_mode` on the authenticated user's project membership:

- `FULL_ACCESS`: total, spent, and contribution progress.
- `SUMMARY_ACCESS`: total and contribution progress, no full breakdown.
- `CONTRIBUTION_ONLY`: contribution progress only.
- `NO_ACCESS`: endpoint returns 403.

## Communication

MVP uses WhatsApp manual links generated by the backend. Cloud API can replace the service implementation without changing invite contracts.


## Linked Ceremonies

Linked ceremonies are represented through `project_links`, allowing an introduction and wedding to remain separate tenant projects while sharing views such as committee context. Budgets stay separate by default unless `shared_budget` is intentionally enabled later.

## Vendors

Vendors are now first-class project-scoped records in `vendors`. They do not require user accounts and can store category, contact, status, notes, and an external URL only.

## Supabase RLS

`supabase/migrations/202607030002_rls_policies.sql` enables row-level security and helper functions for membership and role checks. RLS prevents cross-project access at the database layer; backend RBAC still remains required for response shaping, especially budget visibility.
